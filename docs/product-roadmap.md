# DeclarAI 产品演进路线图

> 日期: 2026-06-04 | 基于差距分析 `docs/gap-analysis.md`

## 产品哲学

DeclarAI 不是一个大而全的系统，而是**围绕报关员核心工作流的一系列能力模块**。每个模块可以独立使用、独立交付价值。模块之间通过共享的通用字段池和存储层自然衔接，不强制依赖。

原则：
1. **一个版本只解决一个核心问题**。不给用户堆功能。
2. **每个 Phase 有独立的用户价值**。用户不需要等全部做完才能用。
3. **模块可组合**。用哪个开哪个，不用的不占界面空间。
4. **先纵深后横向**。先把一条完整链路做透（提取→填码→输出），再扩展到业务管理。

## Phase 1: Extract（智能提取） ✅ 已完成

**解决的问题**：报关员拿到一堆单证，手动翻看、抄录字段到申报系统，费时易错。

**交付内容**：
- 多格式文件导入（PDF/DOCX/XLSX/TXT/ZIP）
- AI 通用字段提取（50+ 字段 + 货物明细数组）
- 文件相关性检测（无关文件警告）
- 通用数据质量审核
- 7 种报关单类型视图切换
- 本地文件存储（`~/Documents/DeclarAI/`）

**用户价值**：一份提单+发票+箱单，过去人工翻 10 分钟，现在拖进去 30 秒出结果。

**当前局限**：
- 字段值是中文文本，不是海关代码
- 没有 HS 编码推荐
- 没有格式化输出（只有 JSON）
- 审核是通用的，不是报关专业的
- 没有客户/业务管理

## Phase 2: Smart Fill（智能填码）

**解决的问题**：AI 提取出来的是"一般贸易"、"水路运输"这类中文，但报关单一窗口要填的是"0110"、"2"这类代码。报关员需要记住或查询大量代码表。同时，HS 编码归类是报关员每天耗时最多的工作。

### 2.1 海关代码表

**做什么**：内置中国海关标准代码表。用户在字段旁看到中文 label，背后自动对应标准代码。输入时支持中文搜索、代码直输、下拉选择三种方式。select 字段的值改为 `{ label, code }` 结构，前端显示中文，数据结构里同时存中文和代码。

**范围**：
- 监管方式代码（0110 一般贸易、0214 来料加工等）
- 征免性质代码（101 一般征税等）
- 成交方式代码（1=CIF、2=CFR、3=FOB 等）
- 运输方式代码（2=水路、3=铁路、4=公路、5=航空）
- 币制代码（USD=502、EUR=300、JPY=116 等）
- 关区代码（4位数字，常用口岸）
- 包装种类代码
- 港口代码（UN/LOCODE）

**实现方式**：代码表作为静态 JSON 文件，存储在 `src/main/code-tables/`，不需要数据库。代码表文件来源为中国海关总署公开资料，由人工整理维护。

### 2.2 HS 编码推荐

**做什么**：输入商品名称（中文），AI 推荐最可能的 HS 编码（4位/6位/10位）。推荐结果附带该编码的：法定单位、关税税率、增值税率、监管条件（是否需要许可证等）、申报要素清单。

**实现方式**：
- 不内置关税税则数据库（数据量大且频繁更新）
- 通过 AI（DeepSeek）根据商品名称 + 已知上下文查询推荐编码
- 或者在系统 prompt 中注入常用章节的 HS 编码知识
- 后续可接入专业的 HS 归类 API

**不做什么**（本 Phase 不做）：
- 不做完整的关税税则查询系统
- 不做税率实时更新
- 不做监管证件的自动申请

### 2.3 申报要素补全

**做什么**：确定 HS 编码后，对该编码对应的商品，提示需要填写哪些"申报要素"（品牌、型号、材质、用途、成分含量等）。这些要素在单一窗口是必填的，缺少直接退单。

**实现方式**：HS 编码关联申报要素清单（同样来自海关公开资料），以提醒形式展示在货物明细中。

## Phase 3: Output（模板输出）

**解决的问题**：AI 提取 + 智能填码完成后，数据还在软件里。报关员需要把它"搬"到单一窗口。如果输出格式和单一窗口的录入顺序一致，抄录效率翻倍。

### 3.1 格式化抄录视图

**做什么**：按照每种单据类型在单一窗口中的字段排列顺序，生成一个**抄录视图**。字段按单一窗口的页面分组排列（表头→表体→表尾），报关员从上到下、从左到右依次抄录。每个字段显示：代码值 + 中文说明。可一键复制单个字段值。

**不做什么**：
- 不做单一窗口 API 自动申报（不合规且风险极高）
- 不做 PDF 表单自动填充

### 3.2 XLSX 导出

**做什么**：将申报单数据导出为 Excel 文件。按照每种单据类型的模板格式，字段映射到对应单元格。支持批量导出（选中多票一次导出）。

**实现方式**：SheetJS 写入 xlsx。每种单据类型一个 Excel 模板配置（字段→列/行映射）。

### 3.3 导出留存在 `exports/`

所有导出文件自动保存在对应业务文件夹的 `exports/` 子目录中，不另存一份。

## Phase 4: Verify（专业审核）

**解决的问题**：AI 填完数据后，报关员需要检查数据质量。通用检查不够，需要报关专业知识驱动的审核。

### 4.1 单证交叉一致性

**做什么**：对比同一票业务的所有导入文件，检查数据是否一致。从发票、箱单、提单、合同中提取的数字（金额、件数、重量）互相对比，列出差异。

**实现方式**：提取阶段为每个文件的提取结果打上来源标记。审核阶段对比不同来源的同一字段值。不依赖 AI——这是确定性规则。

### 4.2 报关逻辑校验

**做什么**：用报关专业知识规则检查数据：
- 净重 ≤ 毛重
- 总价 = 单价 × 数量（近似）
- CIF 价 ≥ FOB 价
- 成交方式与运费/保险费的逻辑关系
- HS 编码与商品名称的合理性（电子产品不会归类到第 2 类植物产品）
- 启运港/目的港与运输方式的逻辑一致性

**实现方式**：规则引擎（硬编码业务规则）+ AI 辅助判断（模糊项）。

### 4.3 单证完整性检查

**做什么**：根据贸易方式和货物类型，判断应该有哪些单证。缺少关键单证时给出提醒。例如：一般贸易进口必须有合同、发票、装箱单、提单。涉及法检商品必须有通关单。

## Phase 5: Desk（业务工作台）

**解决的问题**：报关员同时处理多个客户的多票业务。需要一个超越单票视角的工作台来管理业务流。

### 5.1 客户维度

**做什么**：申报单关联客户。客户有名称、联系方式、常用贸易信息等。按客户筛选、分组查看申报单。客户信息可设为申报单字段的默认值（常用收货人、贸易方式等）。

**实现方式**：新增 `clients` 表和简单的客户管理界面。关联到 `declarations.client_id`。最小化实现，不做 CRM。

### 5.2 申报状态流

**做什么**：真实的状态机：`草稿 → 待申报 → 已申报 → 海关审核中 → 退回补证 → 已修改 → 重新申报 → 放行`。每个状态记录时间戳和备注。状态变更日志。

**实现方式**：`declarations` 表增加 `status` 枚举值和 `status_log` JSON 字段。

### 5.3 工作台仪表盘

**做什么**：首页看板。按状态分栏显示申报单（类似 Trello 的看板视图）。显示本周待处理数、被退单数、即将超期的申报单。按客户、按时间筛选。

### 5.4 历史复用

**做什么**：同一客户的重复业务（同样的商品、HS 编码、贸易方式），自动建议复用上一票的数据。新建申报单时可选择"基于历史申报单创建"。

## 不做的事情（明确边界）

以下功能不在路线图中，明确不做：

- 单一窗口 API 直连自动申报（法律风险）
- 关税税则完整数据库（更新频繁，维护成本高）
- 在线支付关税
- 多用户协作/云端同步
- OCR 图片 PDF 识别（等用户有强需求再说）
- 移动端 App
- 报关员资格考试题库
- 物流追踪
- 货代管理
- 财务结算
