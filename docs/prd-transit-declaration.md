# PRD: 转关运输申报单制单模块

## Problem Statement

当前转关运输申报单（`transit_transport`）仅定义了 10 个基础字段和 9 个货物明细列，与实际业务需求差距很大。完整的转关单需要涵盖基本信息、前程提运单信息、集装箱明细、商品明细以及集装箱与商品的对应关系。此外，海关代码表、申报单位等基础数据缺少维护入口，单据导出只有 JSON 格式，无法交付给客户确认。

## Solution

深度重构转关运输申报单模块，构建四个数据 Section + 基础数据维护 + 模板预设 + 双格式导出的完整工作流：

1. **基本信息**（11 字段）：申报地海关、进出口岸、申报单位、运输方式与工具、承运单位、预计运抵日期等
2. **提运单信息**（10 字段）：前程运输的船舶/航次/提单/收货人等海运信息
3. **集装箱信息**（数组）：箱号、尺寸、关锁、封志等，支持多行录入
4. **商品信息**（数组）：HS 编码、品名规格、数量单位、价格币制等，每行关联集装箱号
5. **基础数据维护**（设置页）：海关代码表、企业信息库，内置数据 + 用户可增删改
6. **模板预设**：保存完整申报模板（预设基本信息+提运单固定字段），新建时一键套用
7. **导出**：Excel + PDF 打印格式，排版美观，适合客户确认

## User Stories

### 申报单录入

1. 作为制单员，我希望在一个页面内分段填写基本信息、提运单信息、集装箱信息和商品信息，以便高效完整地录入转关申报单。
2. 作为制单员，我希望申报地海关和进出口岸从预置代码表中选择（支持搜索和手动输入），以便准确快速填写口岸信息。
3. 作为制单员，我希望申报单位（统一社会信用代码+海关10位编码+企业名称）从预设企业库中自动填充，并且可以在制单页切换企业，以便处理多主体报关场景。
4. 作为制单员，我希望境内运输方式从下拉列表选择（铁路/公路/航空/水路），以便快速完成标准字段填写。
5. 作为制单员，我希望必填字段有明确的视觉标识（红色星号），提交前校验并提示缺失项，以便避免提交不完整的申报单。
6. 作为制单员，我希望集装箱信息以可编辑表格形式录入，每行一个集装箱，支持添加/删除行，以便灵活管理不同数量的集装箱。
7. 作为制单员，我希望每个集装箱行记录箱号、尺寸（20尺/40尺/45尺下拉）、关锁个数和关锁号等完整信息，以便满足海关申报要求。
8. 作为制单员，我希望商品信息以可编辑表格形式录入，每行一个商品，支持添加/删除行，以便灵活管理不同数量的商品。
9. 作为制单员，我希望每个商品行可以在"所在集装箱号"字段中填入一个或多个集装箱号（逗号分隔），以便建立集装箱-商品对应关系。
10. 作为制单员，我希望基本信息中的必填字段（申报地海关、进出口岸、申报单位、境内运输方式、境内运输工具名称、承运单位、预计运抵指运地日期）明确标注且空值时提醒，以便确保申报单完整性。
11. 作为制单员，我希望提运单信息中的必填字段（运输方式、船舶英文名、航次号、提单号、件数、重量、收货人）明确标注且空值时提醒。

### AI 提取与模板

12. 作为制单员，我希望导入单证后 AI 能自动提取转关单的四个 Section 数据，以便减少手工录入。
13. 作为制单员，我希望 AI 提取后能展示待确认项（置信度低的字段），我可以逐一确认或修改，以便信任提取结果的准确性。
14. 作为制单员，我希望可以保存当前申报单的完整内容为模板（基本信息+提运单信息的固定部分），以便下次新建时快速套用。
15. 作为制单员，我希望新建申报单时可以选择已有模板，自动填充固定字段，然后只需导入单据提取集装箱和商品信息，以便大幅提升录入效率。
16. 作为制单员，我希望模板可以命名、编辑和删除，以便管理不同的运输路线和客户组合。

### 基础数据维护

17. 作为管理员，我希望在设置页维护海关关区代码表（关区代码、关区名称、所属关区），以便申报地海关和进出口岸能够准确选择。
18. 作为管理员，我希望海关代码表支持添加、编辑、删除和搜索，初始提供常用口岸的预设数据，以便快速上手和维护。
19. 作为管理员，我希望在设置页维护申报单位信息库（统一社会信用代码、海关10位编码、企业名称、企业简称），以便在制单时快速选择。
20. 作为管理员，我希望可以设置一个默认申报单位，以便新建申报单时自动填充最常用的企业信息。
21. 作为管理员，我希望基础数据（海关代码表和企业库）存储在本地并可导出备份，以便数据安全和迁移。

### 导出

22. 作为制单员，我希望将完整的转关申报单导出为 Excel 文件，排版清晰，每 Section 独立 sheet 或区域，以便客户审阅和数据归档。
23. 作为制单员，我希望将完整的转关申报单导出为 PDF 文件，格式美观专业，包含表头和落款区域，以便正式提交客户盖章确认。
24. 作为制单员，我希望 Excel 导出保留完整的四个 Section + 集装箱-商品对应关系，以便客户可以验证每箱装载的商品。

### 数据一致性

25. 作为制单员，我希望商品信息中关联的集装箱号能自动校验是否在集装箱列表中，以便避免引用不存在的集装箱号。
26. 作为制单员，我希望集装箱信息中填写的总件数和总重量与商品明细的累计值保持一致提示，以便避免数据矛盾。

## Implementation Decisions

### 数据模型扩展

**新增数据库表：**

- `customs_offices` — 海关关区代码表
  - `code` TEXT PRIMARY KEY（关区代码）
  - `name` TEXT NOT NULL（关区名称）
  - `parent_name` TEXT（所属关区）
  - `created_at` TEXT

- `declaration_enterprises` — 申报单位/企业信息库
  - `id` TEXT PRIMARY KEY
  - `credit_code` TEXT（统一社会信用代码）
  - `customs_code` TEXT（海关10位编码）
  - `name` TEXT NOT NULL（企业名称）
  - `short_name` TEXT（企业简称）
  - `is_default` INTEGER DEFAULT 0
  - `created_at` TEXT

- `declaration_templates` — 申报单模板
  - `id` TEXT PRIMARY KEY
  - `name` TEXT NOT NULL（模板名称）
  - `template_data` TEXT NOT NULL（JSON，存储基本信息+提运单的固定字段值）
  - `created_at` TEXT

**扩展现有表：**

- `declaration_files` / `declaration_outputs` — 新增 `output_type` 值：
  - `transit_excel`（转关单Excel）
  - `transit_pdf`（转关单PDF）

**转关单 JSON 数据结构扩展（`declaration.json`）：**

当前 UniversalDeclarationData 保持不变，但 fields 中的字段映射将大幅扩展。新增字段清单见各 Section 定义。

### 模块划分

**模块 1：海关代码表管理（Settings）**
- 位置：Settings 页面新增「基础数据」卡片组
- 子模块 A：海关关区代码管理（表格 + 增删改查）
- 子模块 B：申报单位管理（表格 + 增删改查 + 设默认）
- 预设数据：内置常见口岸（天津、二连、满洲里、阿拉山口等 20-30 条）和企业模板
- 数据持久化到 SQLite

**模块 2：转关单制单表单（Workspace 扩展）**
- Section ① 导入单证（复用现有 FileDropZone）
- Section ② AI 提取结果（复用现有Review Issues）
- Section ③ 基本信息 + Section ④ 提运单信息 — 采用现有三列网格布局
- Section ⑤ 集装箱信息 — 采用 CargoDetailsTable 风格的可编辑表格
- Section ⑥ 商品信息 — 扩展的 CargoDetailsTable，增加「所在集装箱号」列
- 每个商品行的「所在集装箱号」字段支持输入集装箱号（逗号分隔多个），输入时校验是否存在

**模块 3：模板管理（Workspace 头部 + 新建流程）**
- 保存模板：当前申报单 → 提取固定字段 → 命名保存
- 加载模板：新建申报单 → 选择模板 → 自动填充基本信息+提运单
- 模板列表：下拉菜单展示，支持重命名和删除

**模块 4：导出功能（IPC + 后端生成）**
- Excel 导出：使用 xlsx 库（已有），多 sheet 结构
- PDF 导出：使用 pdf-lib 或 puppeteer 渲染 HTML 转 PDF
- Excel 布局：基本信息 Sheet + 提运单 Sheet + 集装箱明细 Sheet + 商品明细 Sheet
- PDF 布局：A4 纵向，表头 + 四段数据 + 落款（申报单位盖章处）

### 集装箱-商品对应关系（方案 A）

选择方案 A：每个商品行增加「所在集装箱号」字段（`container_numbers`），存储逗号分隔的集装箱号字符串。校验逻辑：提交时检查所有引用的集装箱号是否出现在集装箱信息列表中。

### 预设数据

- 海关关区代码：内置约 30 个常用口岸（全国直属海关+主要陆路口岸）
- 申报单位模板：内置 0 条（企业自行添加），首次使用时引导配置

### IPC 通道新增

- `data:customs-offices:list` — 获取海关代码表
- `data:customs-offices:save` — 保存/更新海关代码
- `data:customs-offices:delete` — 删除海关代码
- `data:enterprises:list` — 获取企业列表
- `data:enterprises:save` — 保存/更新企业
- `data:enterprises:delete` — 删除企业
- `data:enterprises:set-default` — 设置默认企业
- `data:templates:list` — 获取模板列表
- `data:templates:save` — 保存模板
- `data:templates:delete` — 删除模板
- `export:transit-excel` — 导出转关单 Excel
- `export:transit-pdf` — 导出转关单 PDF

### UI 交互细节

- 申报地海关/进出口岸：autocomplete 输入框，输入时下拉搜索结果，支持自由文本（代码表未覆盖时）
- 申报单位：下拉选择框，显示「企业简称（海关编码）」，旁边显示当前选中的完整信息
- 运输方式：单选下拉（铁路/公路/航空/水路）
- 集装箱尺寸：下拉（20尺/40尺/45尺/其他）
- 币制：下拉预选常见币种（人民币/美元/欧元/日元等）
- 日期字段：日期选择器（浏览器原生 date input）
- 表格字段：参照 CargoDetailsTable 的交互模式

## Testing Decisions

- 测试内容：数据库迁移（新表创建）、海关代码/企业 CRUD 操作、模板保存与加载、字段必填校验、集装箱-商品对应关系校验
- 不测试内容：AI 提取准确率、PDF 渲染像素级精确度
- 测试风格：延续现有 vitest + SQL.js 内存库模式，React 组件使用 testing-library

## Out of Scope

- 对接海关系统（XML/EDI 申报）
- 电子口岸一键申报
- 报关单状态追踪
- 客户管理和权限
- 在线协作/多人编辑
- 手机端适配
- 电子签名/盖章图片嵌入 PDF

## Further Notes

- 本模块是对现有 `transit_transport` 类型的深度扩展，将替代当前的 10 字段简化版
- 海关代码表初始数据来源：海关总署公告的关区代码表
- PDF 导出初始版本可使用 HTML 模板 + Electron printToPDF 方案，排版迭代可后续优化
- 模板功能的核心价值在于「固定字段复用」：同一客户、同一路线的转关单，收发双方和口岸基本不变，集装箱和货物每次不同
